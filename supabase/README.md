# KiraTakipPro — Supabase Server-Side Foundation

Server-side schema, migrations, RLS policies, SECURITY DEFINER RPCs, and pgTAP tests.
No Electron client code lives here. No secrets belong here.

---

## Prerequisites

- [Supabase CLI](https://supabase.com/docs/guides/cli) ≥ 1.150
- Docker Desktop running
- Node.js ≥ 18 (for future JS integration tests)

---

## Local Supabase stack

> **Non-default ports — intentional.**
> KiraTakipPro uses ports 55321–55324 instead of the Supabase defaults (54321–54324).
> The defaults are already occupied by the AdminPilot local Supabase stack.
> Using a separate port range avoids conflicts when both projects run simultaneously.
>
> | Service     | URL / Address               |
> |-------------|-----------------------------|
> | Project URL | http://127.0.0.1:55321      |
> | Database    | 127.0.0.1:55322             |
> | Studio      | http://127.0.0.1:55323      |
> | Mailpit     | http://127.0.0.1:55324      |
> | Shadow DB   | 127.0.0.1:55320             |

```bash
# Start local stack (Postgres + Auth + Storage + Studio)
npx supabase start

# Studio UI:    http://127.0.0.1:55323
# DB:           postgresql://postgres:postgres@127.0.0.1:55322/postgres
# Project URL:  http://127.0.0.1:55321
```

## Apply migrations

```bash
# Apply all pending migrations to local DB
supabase db reset          # wipes DB, re-runs all migrations + seed.sql

# Or push only new migrations (does not reset)
supabase db push
```

Migration order is enforced by filename timestamp prefix (`20260612000001_` … `20260612000010_`).
Never rename migration files after they have been applied to any environment.

---

## Run tests

```bash
# Run all pgTAP SQL tests in supabase/tests/
supabase test db
```

Tests run inside transactions and ROLLBACK — they leave the DB state unchanged.

## Deploy to remote environments

Three projects: dev / staging / prod.

```bash
# Push migrations to a remote project
supabase db push --project-ref <PROJECT_REF>
```

Environment variables for each Electron build (injected at CI build time, never stored in repo):

| Variable               | Where used                          |
|------------------------|-------------------------------------|
| `SUPABASE_URL`         | Electron main process only          |
| `SUPABASE_ANON_KEY`    | Electron main process only          |

The **service-role key** must never appear in any of these variables in the Electron build.
It belongs only in CI environment variables for migration scripts, and only in the migration runner context.

## Security rules — READ BEFORE TOUCHING ANYTHING

### Service-role key
- **NEVER** place the service-role key in Electron (main process, preload, renderer, asar archive).
- **NEVER** commit the service-role key to this repo in any file.
- The service-role key is only needed to run `supabase db push` in CI. It stays in CI secrets.

### Anon key
- Safe to bundle in Electron. RLS is the actual security layer.
- The anon key grants zero access without a valid user JWT + matching RLS policy.

### Tokens
- User access token → main process memory only, never written to disk.
- Refresh token → OS keychain via `safeStorage` (`userData/cloud-auth.enc`).
- Renderer never receives the access token or refresh token.

### Machine fingerprint
- Never uploaded. The `devices.id` column is `uuid` type with a hyphenated UUID CHECK constraint.
  A 64-char hex sha256 fingerprint physically cannot be stored there.

### Signed URLs
- Snapshot and backup signed URLs are generated by the Electron main process using the
  Supabase JS client (`storage.from(bucket).createSignedUrl(path, expiresIn)`) with the user JWT.
- No service-role key is needed for signed URL generation.

## Schema overview

All tables live in the `ktp` schema. RLS is enabled default-deny on every table.
All privileged writes go through SECURITY DEFINER RPCs.

| Table                   | Purpose                                      |
|-------------------------|----------------------------------------------|
| `ktp.accounts`          | Mirrors `auth.users`; populated by trigger   |
| `ktp.workspaces`        | Tenant workspace; `local_workspace_id` maps Electron DATA.workspaceId; one lock row always exists |
| `ktp.workspace_members` | Membership with `member_role`                |
| `ktp.invite_tokens`     | Time-limited invite links with `token_role`  |
| `ktp.devices`           | Registered app instances (UUID, never hash)  |
| `ktp.workspace_snapshots` | Immutable sync revision log               |
| `ktp.workspace_locks`   | One row/workspace; CAS serialization point   |
| `ktp.cloud_backups`     | Cloud backup metadata (`backup_trigger`)     |
| `ktp.audit_logs`        | Append-only; REVOKE prevents client writes   |

Storage buckets: `ktp-snapshots`, `ktp-backups` — both private, 100 MiB limit.

## Verify packaged app has no service-role key

After building the Electron app:

```bash
# Windows PowerShell
asar extract app.asar C:\tmp\inspect
Select-String -Path "C:\tmp\inspect\*" -Pattern "service_role" -Recurse
# Must return zero matches
```
